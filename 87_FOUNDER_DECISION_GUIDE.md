# 87_FOUNDER_DECISION_GUIDE

## Purpose

This guide helps the founder make build decisions without getting lost in the full ecosystem.

---

# The Main Question

For any new feature, ask:

“Does this help prove the Memory → Blueprint → Visualizer → Build Sheet loop?”

If yes, consider it.

If no, save it for later.

---

# Build Decision Rules

## Rule 1 — Protect the Core Loop

Do not distract from:
- Memory Weaver
- Blueprint Compiler
- Placement Engine
- Design Studio
- BuildBloom

---

## Rule 2 — Avoid Future-System Overload

If a feature belongs to:
- Marketplace
- Moodoor
- Adaptive Intelligence
- Enterprise
- Integrations

then it is probably not MVP.

---

## Rule 3 — Deterministic First

If a feature makes placement random or AI-only, reject it.

---

## Rule 4 — Visualizer Matters

The visualizer is not decoration.

It is the bridge between engine and user trust.

---

## Rule 5 — Build for Proof, Not Perfection

The MVP does not need:
- perfect photorealistic rendering
- every floral asset
- all advanced scoring systems

It needs:
- a believable, structured, saveable, manufacturable blueprint.

---

# Founder Priority Order

1. Get the app running.
2. Get blueprints saving.
3. Get Memory Weaver generating.
4. Get deterministic placement working.
5. Get visualizer working.
6. Get BuildBloom output working.
7. Get 5 beta users to try it.

---

# The Product Test

If a wreath seller says:

“I can use this to plan a design and build it,”

then the MVP is working.
